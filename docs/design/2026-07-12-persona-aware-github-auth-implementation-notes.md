# Implementation Notes: Persona-Aware GitHub Auth for gx

Design doc: `docs/design/2026-07-12-persona-aware-github-auth.md`

## Phase 1: Config surface

### Design decisions
- Added `token_env: Option<TokenEnvConfig>` to `GithubConfig` (`src/config.rs`), `#[serde(rename = "token-env")]`, and updated `GithubConfig`'s hand-written `impl Default` to set `token_env: None` â€” matches the doc's API Design block exactly.
- Added `TokenEnvConfig` (`src/config.rs`) with `derive(Debug, Clone, Deserialize, Serialize, Default)`, `#[serde(default, deny_unknown_fields, rename_all = "kebab-case")]`, fields `default_env: Option<String>` (`#[serde(rename = "default")]`) and `by_org: BTreeMap<String, String>`. Empty by default per the doc â€” the `tatari-tv` -> work classification floor is explicitly NOT encoded here; it belongs to the Phase 2 resolver.
- Added `Config::token_env()` accessor (`src/config.rs:token_env`) mirroring `pr_body_template()` exactly: `self.github.as_ref().and_then(|g| g.token_env.clone()).unwrap_or_default()`, so an absent `github:` or `github.token-env:` block yields the empty `TokenEnvConfig::default()` rather than `None`.
- `gx.yml`: removed the `token-path:` example line; added a COMMENTED `github.token-env` doc block (verbatim from the design doc's YAML example) under a new `github:` section, placed after `logging:` and before the existing `mcp:` block.
- Tests added to `src/config/tests.rs` (not an inline `mod tests` block, per `rust.md` test placement): `test_token_env_block_round_trips`, `test_token_env_unknown_field_fails_loudly`, `test_token_env_absent_yields_empty_default`. No env-touching needed (Phase 1 does zero env reads), so none of these use `env_lock()`.

### Deviations
- **`WORK_TOKEN_ENV` / `HOME_TOKEN_ENV` consts NOT added in this phase.** The doc's Phase 1 bullet says "your call" on placement, but the deeper issue is timing: these two consts have zero callers until Phase 2's `resolve_token_env` exists. This repo's `gx` package compiles BOTH a `lib` target (`src/lib.rs`, `pub mod config` etc.) and a `bin` target (`src/main.rs`, `mod config` etc.) from the *same* source files as two independent crate-graphs. `cargo clippy --all-targets -- -D warnings` (the exact command `otto ci`'s `check` task runs) flags any item unreachable from the `bin` target's own `main()`-rooted graph as dead code â€” reachability from tests (`#[cfg(test)] mod tests`, which is compiled out of the plain, non-test `bin` build) does not save it, and `rust.md` bans `#[allow(dead_code)]` outright. Verified empirically: `cargo check --lib` alone is silent (pub items are the library's external API, not dead), but `cargo check --bin gx` warns, and clippy's `-D warnings` turns that into a hard failure. Adding the two consts now with no caller would have reproduced this exact failure with no way to close it inside Phase 1's boundary (can't touch `resolve_token_env`/`read_token`/`gh_command`). Per the phase's own guidance for this class of problem ("prefer NOT adding dead consts"), they are deferred to Phase 2, where `resolve_token_env` will define and consume them immediately in the same commit.
- **`Config::token_env()` accessor hit the identical dead-code problem** (it is a required Phase 1 deliverable, unlike the consts, so deferring it was not an option â€” the phase's own success criteria require calling it directly on `Config::default()`). Resolved it with a real, in-scope, non-forbidden connection instead of gaming the lint: `Config::load_from_file` (`src/config.rs`) now emits a `debug!` log line reporting the effective `token-env` override state (the `default` value and `by-org` entry count, never the org->env-var contents beyond that) right after a successful parse. This is genuine, useful diagnostic output for the new config surface (consistent with `logging.md`'s "every function tells its story" rule and the existing `log::info!("Loaded config from: ...")` line immediately above it) â€” it does not touch `read_token`, `gh_command`, or any resolver logic, and it is the reason the accessor is no longer dead code in the `bin` target. Recorded here as a deviation because the design doc's API Design section did not call out this log line explicitly.

### Tradeoffs
- Wiring the accessor into a diagnostic log line (chosen) vs. wiring it into `gx doctor` output (rejected): `doctor.rs` currently has zero awareness of `GithubConfig`, and the design doc never mentions `doctor` â€” adding that surface would be scope creep beyond "Config surface." The load-time debug log is strictly narrower and stays entirely inside `config.rs`.
- Kept `TokenEnvConfig` fields exactly as specified (`default_env` / `by_org`) rather than a richer per-org struct â€” the doc is explicit this is an overrides-ONLY map, and any richer shape would pre-empt Phase 2 design choices not yet made.

### Open questions
- None.

## Phase 2: Persona resolver

### Design decisions
- Placed the resolver in a NEW `src/persona.rs` seam (not `src/github.rs`) â€” `pub fn resolve_token_env(org, config) -> Result<String>` â€” with `WORK_TOKEN_ENV = "GITHUB_PAT_WORK"`, `HOME_TOKEN_ENV = "GITHUB_PAT_HOME"`, and a third `GH_PERSONA_ENV = "GH_PERSONA"` const, plus a private `Persona { Work, Home }` enum with `env_name()` so the two literal names live once each and can't drift (design doc "Why classification-in-code" note). Rationale for a dedicated module: keeps `github.rs` focused, gives the persona logic its own `tests.rs`, and the doc explicitly lists `src/persona.rs` as an acceptable seam.
- `GH_PERSONA` parsing (`gh_persona()` in `src/persona.rs`): trims the raw value; trimmed-empty -> `None` (fall through); exact `work`/`home` -> the persona; ANY other non-empty value -> loud `bail!` naming `GH_PERSONA`. This is the intentional stricter divergence from the `gh()` shell wrapper (which falls back to `$PWD` on a bogus value) required by Resolution model #1 â€” a typo like `wrok` must never silently pick an identity.
- The 5-level precedence lives entirely in `resolve_token_env`: `GH_PERSONA` > config `by_org[org]` > built-in `tatari-tv -> WORK_TOKEN_ENV` > config `default_env` > built-in `HOME_TOKEN_ENV` floor. The `tatari-tv -> work` classification is a CODE floor above the config `default`, so a config `default` can only override the HOME floor and can never shadow the work classification (`test_default_applies_only_to_non_tatari_tv` pins both halves).
- Uses the `Config::token_env()` accessor from Phase 1 (yields the empty `TokenEnvConfig` default when the block is absent), never the raw `github.token_env` field.
- Function-level debug logging per `logging.md`: entry logs `org`; each return logs `org -> {NAME} (via ...)`. Only env-var NAMES are logged, never a token value (the resolver never reads a value).
- Tests in `src/persona/tests.rs` (2018+ submodule style, `#[cfg(test)] mod tests;` in `persona.rs`), not an inline block. All 9 cover the FULL precedence table (built-ins both ways, by-org beats built-in tatari-tv, by-org selects the service PAT, GH_PERSONA work beats by-org, GH_PERSONA home beats built-in tatari-tv, config default applies only to non-tatari-tv, bogus GH_PERSONA -> Err, empty GH_PERSONA treated as unset). Every test mutates `$GH_PERSONA` through a `with_gh_persona()` helper that acquires the shared `crate::test_utils::env_lock()` and saves/restores the prior value (rust.md env-test contract). The GH_PERSONA-beats-by-org test is the required BITE test: it maps `scottidler -> GITHUB_PAT_HOME` in by-org and asserts `GH_PERSONA=work` still wins, so an inverted precedence (by-org checked first) fails it.

### Deviations
- **Module wired into the LIB target only (`pub mod persona;` in `src/lib.rs`), NOT into `src/main.rs`.** This is the correct Phase-2 state and the resolution of the exact dead-code footgun Phase 1 documented. gx compiles both a `lib` and a `bin` target from the same sources as independent crate-graphs; `cargo clippy --all-targets -- -D warnings` (otto's `check` task) flags any item unreachable from the `bin`'s `main()`-rooted, non-test graph as dead code, and `rust.md` bans `#[allow(dead_code)]`. Verified empirically: an unused `pub fn` in a module declared in `main.rs` fails the bin build (`function ... is never used`), whereas the same `pub fn` in a module declared ONLY in `lib.rs` is treated as the library's external API and is exempt â€” and the bin, not declaring the module, has nothing to flag. Since Phase 2's whole point is a standalone resolver with NO production caller yet (Phase 3 rewires `read_token` to call it), lib-only wiring is the honest way to land it green without a fake caller or a suppressed lint. Phase 3 will add `mod persona;` to `src/main.rs` when `read_token` (in `github.rs`, compiled into both targets) starts calling `crate::persona::resolve_token_env`, at which point it becomes reachable from `main()` and non-dead in the bin.
- `resolve_token_env` is `pub` (the doc's API-Design sketch shows a private `fn` in a github.rs context). `pub` is required for the lib-target dead-code exemption above and is a legitimate visibility for the gx lib's public surface. Consts stay module-private (used by the reachable pub fn, so not dead).
- Added `GH_PERSONA_ENV` as a named const for the env-var name (not in the doc's const list). `logging.md`/`rust.md` "no magic strings for the same literal used twice" â€” it is referenced by `gh_persona()` and by two debug lines.

### Tradeoffs
- Dedicated `src/persona.rs` module (chosen) vs. putting `resolve_token_env` + consts inside `src/github.rs` (the doc's other option): a private `fn` in `github.rs` would have hit the identical bin dead-code failure (github.rs is declared in main.rs), forcing either a fake caller or a suppressed lint this phase. The lib-only module sidesteps that cleanly and gives the resolver its own focused test file.
- Case-sensitive exact `work`/`home` match for `GH_PERSONA` (chosen) to mirror the `gh()` wrapper's shell `case` exactly (the doc's stated intent: "accepted VALUES stay work|home to match the wrapper"), rather than the case-insensitive treatment `cli.md` prescribes for CLI enum flags. `GH_PERSONA` is an env var mirroring a shell wrapper, not a CLI flag; a mis-cased `WORK` is a loud error (never a silent wrong identity), which is fail-closed and acceptable.

### Open questions
- **Pre-existing flaky test `lock::tests::test_lock_reacquirable_after_holder_drops` (`src/lock/tests.rs:77`), NOT caused by this phase.** It intermittently fails ONLY under `otto ci`'s concurrent `check`+`test` load (the reacquire of a dropped `File::try_lock`/flock returns `WouldBlock` under heavy CPU/FS contention); it passed 3/3 under plain `cargo test --workspace` including all persona tests, and the final `otto ci` gate for this commit was green. A pure, lib-only resolver reading only `$GH_PERSONA` + config (its unit tests touch no filesystem and no flock) cannot mechanically affect a flock reacquire. Out of scope for Phase 2 to fix (it lives in the lock module, unrelated to persona auth); flagged here for the parent â€” the lock test should be hardened against parallel-load flakiness (`taste.md`: flaky tests get hardened, not retried), likely as its own targeted fix.
